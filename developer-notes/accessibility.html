Accessibility

Accessibility is the practice of making your websites usable by as many people as possible. We traditionally think of this as being about people with disabilities, but really, it also benefits other groups such as those using mobile devices, or those with slow network connections.

Alt text

One way to make the elements of your page more comprehensible to screen readers is to provide alt (alternative) text for images. Alt text is provided as an attribute to the <img> element to describe the image to the screen reader. A screen reader will be able to read aloud the description of the image to the user, as opposed to simply informing the user that an image exists on the page.

        

<img src="brown-bear.jpg" alt="A brown bear" />

Links

Descriptive Link Text

Use descriptive link text within your links.

        

<!-- Bad -->
<a href="#">Click here</a>

<!-- Good -->
<a href="#">Information about accessibility</a>

It is worthwhile remembering that the href attribute should not be empty as this then ensures the link will be reachable by keyboard users.

Language Attribute

Declaring a language attribute on the HTML element enables a screen reader to ready out the text with the correct pronunciation.

        

<html lang="en">

<html lang="en-GB">

Headings

It is recommended to use one <h1> element per page.

Heading elements - <h1> to <h6> should be kept in order.

        

<!-- Bad -->
<h1>A header</h1>
<h4>A smaller header</h4>
<h2>Another header</h2>

<!-- Good -->
<h1>A header</h1>
<h2>A smaller header</h2>
<h3>Another header</h3>

Label

The HTML <label> element represents a caption for an item in a user interface. The label text is visually associated with its corresponding text input. A screenreader will read out the label when the user is focused on the form input, making it easier for assistive technology users to understand what data should be entered.

You can click the associated label to focus/activate the input, as well as the input itself.

To associate the <label> with an <input> element, you need to give the <input> element an id attribute. The <label> then needs a for attribute whose value is the same as the input's id.

        

<label for="firstName">First name</label>
<input name="firstName" type="text" id="firstName" />

There is another style to consider. You can next the <input> directly inside the <label> element. In this instance, the for/id attributes are not needed as the association is implicit.

        

<label>First name
    <input name="firstName" type="text" />
</label>

User Container Elements for Layout

Elements like <div> and <span> are for layout. They're semantically meaningless and they don't have keyboard or touch support in any browser. It is recommended to not use a <div> or <span> element for a button when you could use a semantically meaningful <button> element instead.

        

<button>Click Me</button>

Semanic Tags

When you think of styling text on a webpage, you may have noticed some overlap between the following pairs of tags:

        

<b> <strong>
<i> <em>

When you try using any one of these elements, you'll notice that they have similar styling. Both the <b> and <strong> tags bolden text and the <i> and <em> italicize text. However, there is a fundamental difference between each of the tags in a pair when it comes to accessibility.

The <b> and <i> elements simply denote how the text should look. The <strong> and <em> elements however, denote how text should be understood. Each one affects how the screen reader interprets them. Text within these elements is read out with a different voice to indicate the emphasis for each. These elements are known as semantic tags.

ARIA

The difference between HTML5 and ARIA is that ARAI is an add-on of sorts. HTML5 is your actual semantic page code. For example, the <header> HTML5 element is what you'd use for the header of your page, where you mau have your site title, logo, and main navigation.

        

<header> ... </header>

You can do pretty much the same thing with ARIA, at least as far as assistive technology is concerned.

        

<div role="banner"> ... </div>

The nice thing about the HTML5 element is that its semantic HTML. It provides meaning to the structure of the page, a <div> element does not do this. If a browser doesn't support HTML5, but does support ARIA, we can use both. It maybe odd to use HTML5 if some browsers may not support it, but over time, browsers will support HTML5.

It is always recommended to avoid using ARIA attributes, if possible.

        

<header role="banner"> ... </header>

If the page is rendering in an older browser that doesn't know what <header> is, it will still render the element. Browsers render all unknown elements, such as HTML5 elements in older browsers, inline. <header> is a block element in HTML5, so you'll want to use it that way. Give all your HTML5 elements display: block in your CSS stylesheet, to make sure you're covered in older browsers. This can be placed within your reset stylesheet.

Note: IE11 does not recognise the <main> HTML element. So as described, set this element to have the display property of block, and this element will then no-longer be rendered as inline within IE11.

HTML5 provides support for ARIA attributes within the specification. Just ensure your HTML DOCTYPE is set to HTML5.

        

<!DOCTYPE html>

Here are some of the HTML elements and their associated ARIA role attributes:

        

<header role="banner"> ... </header>
<nav role="navigation"> ... </nav>
<footer role="contentinfo"> ... </footer>
<aside role="complementary"> ... </aside>

<article> ... </article>
