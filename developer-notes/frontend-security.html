Frontend Security

Client XSS

On the front end, a Cross-Site Scripting (XSS) vulnerability happens when dynamic data is sent to a user without being validated for malicious content. For example, it would be like sending JavaScript code that can be executed within a form or sent as a post/tweet.

XSS allows malicious code to be executed.

Most often, XSS vulnerabilities occur when websites or applications don't remove executable code from input fields that their users have access to.

Reflected XSS occurs when executable code is injected in an HTTP response. The injected script is not stored on the server, but instead impacts users who are lured into opening a malicious link. When this happens, the injected code travels to the vulnerable application and reflects the attack back to the user's browser. The browser then executes the code because it appears to have come from a trusted server. By using strict input validation, this issue can be avoided.

Cross-site request forgery (CSRF)

CSRF (also referred to as XSRF) is when an attacker uses an HTTP request to access user information from another site on which they are authenticated. An example of CSRF might be changing the login credentials, or making a purchase. A server simply sees the request and performs it. Tokens can be used to resolve this issue as tokens are difficult to spoof, can be sent with the request and validated by the site, which prevents attempts at CSRF. You can also validate origin headers submitted with requests.

SQL Injection

SQL injection is a technique that uses code injection to attack database type applications. Malicious SQL queries are inserted via input from the client to the application. An attacker can disrupt a database, disclose or corrupt data, and many more potential issues. Prepared statements with variable binding can be used to prevent SQL injection. This allows databases to distinguish between data and code, regardless of the input submitted. Whitelisting can be used that specifies what input should look like and doesn't allow input that does not match a specified pattern.

iFrames

It is important to remember that displaying content inside of an iFrame means you're trusting that content not to be vulnerable or malicious. iFrames and XSS attacks usually go hand in hand. If the content inside of the iFrame is coming from your domain, its important to make sure that the iFramed content is secure and not vulnerable to an attack.

It's important to make sure that sensitive pages in your site or application don't allow for use inside of an iFrame. This can be achieved by using the X-Frame-Options http response header.

The X-Frame-Options is an HTTP response header that allows you to define if and how their websites can be loaded into frame elements on other sites. It comes with three options: ALLOW, DENY, and SAMEORIGIN. The latter meaning a page can only be framed by other pages with the same origin - the same domain, URI scheme and port.

Cookies

It is very easy to set a cookie without relying on a server. When working with cookies, it is important to remember that they can be vulnerable to XSS and CORS exploits. Cookies are also not encrypted by default (plain text), meaning that they should not be used to store sensitive details such as passwords.
