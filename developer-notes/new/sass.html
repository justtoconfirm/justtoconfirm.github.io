Sass & SCSS

The new main syntax (as of Sass 3) is known as SCSS (or "Sassy CSS") and is a superset of the CSS3 syntax. This means that every valid CSS3 stylesheet is valid SCSS also. The second, older syntax, is known as the indented syntax. This is named Sass.

Sass was designed by Hampton Catlin and developed by Natalie Weizenbaum.

Sass lets you use features that don't exist in CSS yet like variables, nesting, mixins, and inheritance. A proprocessor will take your preprocessed Sass file and save it as a normal CSS file that you can then use in your website.

SCSS files use the file extension of .scss

Ruby Sass vs LibSass

The original version of Sass was written in Ruby. With Ruby Sass, you install Sass, run the compiler and your files will be compiled into CSS. Ruby Sass will not work if your development environment is outside of Ruby and if a project is large, the Ruby Sass engine can take a bit of time to compile to CSS.

LibSass is a C/C++ port of the original Sass engine that doesn't rely on Ruby to generate CSS, so it can be implemented in other languages.

By itself, LibSass doesn't do anything, it's just a library. For LibSass to work, you'll need a wrapper (or implementor) around LibSass to run the library and compile your style sheets. Some of the most common LibSass wrappers are node-sass, grunt-sass and even a Ruby wrapper is available.

If your project is using node-sass, LibSass is the core library and node-sass is the wrapper that allows compiling inside of Node.

LibSass has the advantage of increased compiling speed and is much faster than Ruby Sass.

LibSass is slightly behind the development curve compared to Ruby Sass, however, when it comes to certain features. For a list of the inconsistencies between LibSass and Ruby Sass, visit http://sass-compatibility.github.io/

Ruby Sass will not release any new features until LibSass catches up. Once this occurs, there will be feature parity between the two moving forward.

Install

Sass can be installed on Windows, Mac, or Linux using NPM (Node Package Manager). This will install the pure JavaScript implementation of Sass which runs slower than other options of Sass and will be installed globally on your machine.

        

$ npm install sass -g

Alternatively, you can install node-sass. A library that provides binding for Node to LibSass, the C version of Sass. This allows you to natively compile .scss files to .css

It's best to install node-sass as a devDependency within your project. This will be listed within your package.json file.

        

$ npm install node-sass --save-dev

The NPM documentation for node-sass can be found at: https://www.npmjs.com/package/node-sass

Within your package.json file, create a script that will compile your SCSS into CSS.

The script below will compile the SCSS file named main.scss within the src/styles directory and output a CSS stylesheet named style.css within the dist/styles directory.

        

{
    "compile:scss": "node-sass --include-path scss src/styles/main.scss dist/styles/style.css",
}

This script can be run from within your terminal.

        

$ npm run compile:scss

Syntax

Partials

You can create partial Sass files that contain little snippets of CSS that can then be included in other Sass files. This is useful to modularize your CSS and help keep things easier to maintain.

A partial is a Sass file named with a leading underscore. You may name it something like _partial.scss with the underscore used to let Sass know that the file is only a partial file and it should not be generated into a CSS file.

We have previously used the @import rule to use a partial, however this is being replaced by the @use rule.

The Sass team discourages the continued use of the @import rule. Sass will gradually phase it out over the next few years and eventually remove it from the language entirely. Prefer the @use rule instead.
