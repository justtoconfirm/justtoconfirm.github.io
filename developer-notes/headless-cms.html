Headless CMS

A headless content management system (CMS) is a backend only content management system built from the ground up as a content repository. The content is accessible via a RESTful API. A Headless Architecture is a phrase used in development to describe an approach which splits the code base between server side (RESTful APIs and web services) which defines data and business logic, from client side code, such as the frontend and JavaScript.

To summarise, a headless CMS is like a traditional CMS but without any way to present the content being created and stored within it. It simply allows for the creation, reading, updating, and deletion (CRUD) of content.

The lack of a front-end means that developers can use any front-end technology they wish to present the content. Many headless CMSs have been purpose built for IoT (Internet of Things) applications.

Using a headless CMS should mean that our front-end will consume the API of the CMS via AJAX requests. The simplest way of doing AJAX in JavaScript is using the XMLHttpRequest object or the Fetch API.

A core feature of Gatsby is its ability to load data from anywhere. This is what makes Gatsby more powerful than many other static site generators. Many CMS's now support a "headless" mode. This allows content creators to manage their content through a familiar admin interface but allows content to be accessed via API endpoints, allowing for a fully customised frontend experience. Gatsby incorporates GraphQL and React to build the frontend.

To simplify this, Gatsby will pull data from the backend "headless" CMS and we can generate static front-ends.

Decoupling Drupal

The concept of 'headless' aims to separate the front-end, the way the site is displayed from Drupal, from the backend of the CMS (Content Management System). What it essentially means is that all the services and the content management are handled by Drupal while the core user experience (UX) or the way the site is displayed in a browser, is controlled via a JavaScript front-end framework such as Angular, or a library such as React. Ths is achieved via Drupal's RESTful API service.

Headless Drupal can be known as a way of using Drupal as a service where the content management and the logic of the site are provided by Drupal and the front-end is done using JavaScript (React, for example).

With Drupal 8, RESTful services have been implemented into the Drupal core. Drupal uses JSON, one of the most widely adopted API formats, for its web services.

Since the logic of the website and the content management is handled by Drupal, redesigning a site visually would only require tinkering with the front-end. This frees up the front-end team to work on the way the site looks, in-turn, freeing up the back-end team to handle the site's backend architecture.

Headless Django

Django is a Python framework and DjangoCMS is a CMS that makes use of this framework. At present DjangoCMS works on the assumption that it will serve its content directly to a web browser. The DjangoCMS team are working on the ability to serve content via RESTful APIs to any suitable system, such as a React or Angular front-end.

Decoupling WordPress

Ever since REST API was bundled in the WordPress core, from version 4.7 back in December 2016, developers can experiment and use it in a decoupled way. WordPress does not follow the traditional Model-View-Controller (MVC) design pattern, instead it is written in a procedural way, using function-based code. WordPress core developers used a system of hooks which allowed other developers to modify or extend certain functionalities.

The WordPress REST API allows you to interact with your WordPress installation by sending and receiving JSON objects.

You can examine the built-in endpoints by going to the /wp-json/ route.
