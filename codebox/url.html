URL

A URL, or Uniform Resource Locator, is a mechanism used by the browser to retrieve any published resource on the web. It is nothing more than the address given to a unique resource. Such resources can be an HTML page, CSS stylesheet, an image or anything else that maybe published.

Scheme

The first part of the URL is the scheme, which indicates the protocol that the browser must use to request the resource. Usually for websites the protocol is HTTPS or HTTP (its unsecured version). The browser can handle other schemes such as mailto: (to open a mail client) or ftp: (to handle file transfers).

    

https:
http:
mailto:
ftp:

Authority

The authority follows the scheme, which is separated by the character pattern :// and includes the domain name, but can also include the port number (separated by a colon).

The domain name indicated which web server is being requested. Usually the domain name is used, but an IP address may also be used.

The port indicates the technical "gate" used to access the resources on the web server. It is usually omitted if the web server uses the standard ports of the HTTP protocol (80 for HTTP and 443 for HTTPS) to grant access to the resources.

    

://www.example.com
://www.example.com:80

The mailto: scheme does not use an authority for the mail client, therefore the colon is not followed by two slashes and only acts as a delimiter between the scheme and the mail address.

    

mailto:contactme@mydomain.com

Path to resource

The path to resource is a path that represents the physical location of a file on a web server.

    

/path/to/myfile.html

Anchor

An anchor represents a sort of "bookmark" inside the resource giving the browser the directions to show the content located at that "bookemarked" spot. Within an HTML document, the browser will scroll to the point where the anchor is defined. The hash (#) is also known as the fragment identifier and is never sent to the server with the request.

    

https://www.example.com/path/to/myfile.html#anAnchor

URL Parameters

URL parameters, also known as "query strings" or "URL query parameters", are elements that are inserted into your URLs to help you filter and organise content, or track information on your site.

URL parameters are a way to pass information about a click using the URL itself and can be used to pass data from page to page within a URL.

If your website has many paramater-based URLs, it's important to signal to the web crawlers which pages not to index and to link to the static, non-parameterised page when considering SEO. A static page URL would look something like this:

    

https://www.domain.com/shoes

Once you've decided on which static page should be indexed, remember to canonicalize it. Set up canonical tags on the parameterised URLs referencing the preferred URL. This will send a signal to crawlers that only the main landing page is to be indexed and not the parameterised URLs.

You'll notice that on this page, the URL of page=url is passed into the URL as a parameter with a value when you click on the link to open this page.

    

<a href="url.html?page=url">URL</a>

The URL parameter refers to the portion of the URL that comes after a question mark (?) and are made of a key and a value, separated by an equal sign (=). Multiple parameters are each then separated by an ampersand (&).

    

https://www.domain.com/page?key1=value1&key2=value2

URL parameters are commonly used to sort content on a page, making it easier for users to navigate products in an online store. These query strings allow users to order a page according to specific filters and to view a set amount of items per page.

    

// Main page
https://www.onlinestore.com/shoes

// URL parameters
https://www.onlinestore.com/shoes?rank=newest
https://www.onlinestore.com/shoes?sessionid=niketrainers

There are two types of URL parameters:

  • Content-modifying parameters (active) - parameters that will modify the content displayed on the page, for example to send a user directly to a specific product.
  • Tracking parameters (passive) - used for advanced tracking, parameters that will pass information about a click, for example which network it came from and/or campaign - this won't change the content on the page.

Here's an example of a content-modifying parameter to send a user directly to a specific product called 'xyz':

    

https://www.domain.com?productid=xyz

Here's an example of a tracking parameter used to track traffic from your newsletter:

    

https://www.domain.com?utm_source=newsletter&utm_medium=email

Or, to collect campaign data with custom URLs:

    

https://www.domain.com?utm_source=twitter&utm_medium=tweet&utm_campaign=summer-sale

URL Parameter Examples

  • Tracking - UTM codes used for tracking traffic from ads and campaigns
  • Reordering - Ordering items according to specific parameters
  • Searching - To find results using a site's internal search engine
  • Identifying - Sorting pages, such as image gallery, by size, category etc.
  • Paginating - Segment content on pages
  • Translating - Adding the language name to the end of a URL string
  • Filtering - Filter distinct fields
    

// Tracking
https://www.domain.com?utm_content=bottom%20cta%20button?utm_source=newsletter123&utm_medium=email?sessionID=32764

// Reordering
https://www.domain.com?sort=price_ascending?order=highest_rated?rank=newest

// Searching
https://www.domain.com?search=search_term?query=search_term

// Identifying
https://www.domain.com?product=small_blue_shoes?category=shoes?productid=xyz

// Paginating
https://www.domain.com?page=2

// Translating
https://www.domain.com?language=en

// Filtering
https://www.domain.com?type=yellow?price_range=20-50

Absolute URLs

In your browser's address bar, a URL doesn't have any context, so you must provide a full (or absolute) URL. You don't need to include the protocol (the browser uses HTTP by default) or port (which is only required when the targeted web server is using an unusual port), but all other parts of the URL are necessary.

When a URL is used within a document, such as in an HTML page, the browser already has the document's own URL, so it can use this information to fill in the missing parts of any URL available inside that document.

If a URL starts with "/", the browser will fetch that resource from the top root of the server.

This is a full absolute URL:

    

https://www.domain.com/en-GB/docs

Implicit protocol - in this case, the browser will call that URL with the same protocol (https:) as the one used to load the document hosting that URL:

    

//www.domain.com/en-GB/docs
