Introduction to Gatsby

Gatsby is a React-based, open-source framework for creating websites and web applications.

It can be used to build:

  • Portfolio websites or blogs.
  • High-traffic e-commerce websites such as a store.
  • Company homepage.

Gatsby wraps together web technology and tools like JavaScript, Git, CI/CD, and APIs (Application Program Interfaces) to provide a modern and quick experience.

Components within Gatsby are built using React and are rendered at build time to the DOM (Document Object Model) as static HTML, CSS & JavaScript files.

A Gatsby site will be a lightning-fast Progressive Web App (PWA) out of the box.

Gatsby is a framework rather than a library because it comes with everything. Routing, data management, build process, and much more is all included.

Gatsby uses a few technologies under the hood:

  • React - A JavaScript library for building user interfaces. It's the library that Gatsby uses to build pages and structure content.
  • GraphQL - A query language that allows you to pull data into your website. It's the interface Gatsby uses for managing site data.

Gatsby uses React under the hood but adds features on top to improve the development and build process. It is not a CMS (Content Management System), there's no backend admin to login to make changes, it is purely for frontend only.

Using React DevTools is a good way to debug your React components and Gatsby site.

You can use ESLint, which includes Prettier under the hood, to check the quality and tidy-up your code during the build. This cab be installed as a plugin for code editors such as VSCode or as an NPM package.

Gatsby has a plugin ecosystem allowing you to install plugins to extend the basic functionality.

Gatsby makes use of Rehydration, where we can have static HTML rendering the pages but use React/JavaScript to add dynamic content, such as a counter, or anything dynamic on the page.

File Structure

Everything must go through Gatsby - images, CSS, fonts, assets etc.

The public directory is where Gatsby will build and deploy our code that will regenerate when building your app. You shouldn't need to make any changes to the contents of this directory.

The static directory is for files that you simply want to be served up, for assets such as logos and favicons. These assets can be referenced by a static path, but it's often better to put all assets and code into the src directory.

Most of our Gatsby application code and assets will sit in the src directory. The src/pages directory is the only folder that is Gatsby specific.

You can add a .gitkeep file into empty directories so that Git doesn't delete the empty directory before you've added something to it when using version control.

An example src file structure (pages you need, everything else can be changed):

    

src /
    assets /
        fonts /
            fontfile.woff2
        images /
            bg.svg
            stripes.svg
    components /
    pages /
    styles /
    templates /
    utils /

SEO (Search Engine Optimisation)

Gatsby makes your site fast and efficient for search engine crawlers, like Googlebot, to crawl your site and index your pages. Because Gatsby pages are server-side rendered, all the page content is available to Googlebot and other search engine crawlers. If you view the source of a page, you will see the fully rendered HTML document.

In June 2021, Google implemented a new ranking factor for site speed, calling the algorithm update the Core Web Vitals (CWV) update.

Adding metadata to pages, such as page title, meta description, alt text and structured data using JSON-LD (JavaScript Object Notation for Linked Data), helps search engines understand your content and when to show your pages in search results.

A common way to add metadata to pages is to add react-helmet components (together with the Gatsby React Helmet plugin for SSR support) to your page components.

Google uses structured data that it finds on the web to understand the content of the page as well as to gather information about the web. Here's an example of a structured data snippet added using react-helmet in the JSON-LD format that may appear on a contact page:

    

<Helmet>
    <script type="application/ld+json">
        {`
            {
                "@context": "https://schema.org",
                "@type": "Organization",
                "url": "https://www.spookytech.com",
                "name": "Spooky technologies",
                "contactPoint": {
                    "@type": "ContactPoint",
                    "telephone": "+5-601-785-8543",
                    "contactType": "Customer Support"
                }
            }
        `}
    </script>
</Helmet>

When using structured data, you'll need to test during development with the Rich Results Test from Google.

An XML sitemap can also be used to help with crawling your website. An XML sitemap is a file that lists a site's important pages, making sure Google can find and crawl them all. It also helps search engines to understand the structure of your site.

React Components

A React component is a function that returns a React element. A React element is an object that React uses to render DOM (Document Object Model) elements. React elements are written using JSX. This is a JavaScript syntax extension that describes the DOM structure for your component and looks a bit like HTML.

A simple React component might look something like this:

    

const Greeting = () => {
    return (
        <h1>Hello World</h1>
    )
}

Gatsby has three types of component:

  • Page Component - contains all the UI (User Interface) elements for a specific page of your site and are located within the src/pages directory.
  • Building-block Component - smaller components that represent parts of the user interface, not the entire page.
  • Pre-built Component - components that are built into the Gatsby library and found in the NPM package. Example includes the <Link> component.

Writing a React component

There are three main steps to writing a React component:

  • Import React from the react NPM package, so that you can use JSX inside your JavaScript file(s).
  • Define your component. It should be a function that returns a JSX element.
  • Export your component, so that it can be imported into other components and be used by other parts of your site.
    

// Step 1: Import React to use JSX inside your JavaScript file
import * as React from 'react'

// Step 2: Define you component. This should always start with an uppercase letter
const MyComponent = () => {
    return (
        <h1>Hi, welcome to my site!</h1>
    )
}

// Step 3: Export your component
export default MyComponent

Your component must return a single React element.

You can use props (properties) in your Reacy components to change how a component renders. You can define your own props when you build a component and React also has some built-in props, like children and className, you can use.

Components with children

When you render a component, the children prop will automatically be passed whatever content comes between the opening and closing tags for that component. This is helpful when you want to create a component that wraps some generic content.

    

import React from 'react'
import Frame from '../components/frame'

const GalleryPage = () => {
    return (
        <Frame>
            <p>This will be passed in as children</p>
        </Frame>
    )
}

export const GalleryPage

Then in the component definition, the children prop will get passed whatever elements come between the opening and closing tag.

    

import React from 'react'

const Frame = ({ children }) => {
    return (
        <>
            <h1>This is the page title</h1>
            { children }
        </>
    )
}

export const Frame

In the browser, the actual DOM (Document Object Model) elements will look something like this:

    

<h1>This is the page title</h1>
<p>This will be passed in as children</p>

Creating a Gatsby site

Now that the Gatsby CLI is installed, we can use it to setup a Gatsby site.

Open the command line and use the cd command to change directories into the folder where you want to create your new Gatsby site. For example, if you want to created your site on your desktop, you would enter:

bash -- 85x24

    $ cd Desktop

Once in your chosen directory/folder, run the new command from within the command line. This will start up the interactive prompt to assist you with creating your Gatsby site.

bash -- 85x24

    $ gatsby new

Once the configuration has completed, you should have a new Gatsby site. For our examples, we will be calling the folder containing the Gatsby project as my-first-gatsby-site.

When you build your project, the name of your project will be set as the title inside the siteMetaData object within the gatsby-config.js file.

Development

Now that the Gatsby site has been installed, we can begin development. First, change into the directory where your site is located. In this example, or Gatsby site is in a folder named my-first-gatsby-site.

bash -- 85x24

    $ cd my-first-gatsby-site

Now from within your site directory, start the development server by running the following command:

bash -- 85x24

    $ gatsby develop

Gatsby has a development server built-in so you can view your site in a browser locally. The development server is a useful tool for working on your site locally. When running, you can interact with a local copy of your site and test changes to your code to make sure they work before publishing.

After a few moments, the command-line should output a message confirming your development server is running and ready to go. Open a web browser and navigate to:

http://localhost:8000

You will be able to visit the site locally using this URL for as long as your development server is running. To stop the development server, go back into your terminal window and hit CTRL + C.

Building & Adding Pages

As you navigate through a Gatsby site, you're not necessarily navigation from page to page, the application is loading the data it needs to render using the components built. This helps to make Gatsby extreamly fast. Gatsby generates the HTML from the data so this helps to make the site load much faster.

Pages can be dynamically or statically generated. Pages are written as functions in React and these are ECMAScript modules meaning that they need to be exported and imported. Gatsby uses hot reloading to update the page during development when changes are made.

All pages within your Gatsby site are located within the src/pages directory of your project. To begin when you first create a project, you will see the index.js as the Homepage and 404.js (404 page) by default.

The pages are built using React components and are known as Page Components. This type of component contains all the UI (User Interface) elements for a specific page of your site, for example the Homepage, About us, Blog etc.

We will now update the Homepage. Open the src/pages/index.js file and replace the default content with the code below:

    

import * as React from 'react'

const IndexPage = () => {
    return (
        <main role="main">

            <title>Homepage</title>

            <h1>Welcome to my Gatsby site!</h1>

            <p>I'm making this by following a Gatsby Tutorial.</p>

        </main>
    )
}

export default IndexPage

Once your file has been saved, the development server will reload and the changes to the homepage will be visible in the browser. You may need to wait a moment whilst the development server rebuilds.

Gatsby automatically creates pages for React components that are the default export of files in the src/pages directory.

If you attempt to visit a URL for a page that doesn't exist, Gatsby will display the 404 page component (src/pages/404.js) to show an error. If you attempt to view the 404 page on localhost, you'll need to click the Preview custom 404 page button on the development 404 page.

When creating a 404 page, the function cannot start with a number, so you may need to name it ErrorPage or FourOhFourPage instead.

The 404 page itself won't show up in development mode, but you can preview it. The page will be updated correctly when we move the site into production and deploy.

Now that the Homepage has been created, we can add an About page. Create a new file named about.js in the src/pages directory.

    

import * as React from 'react'

const AboutPage = () => {
    return (
        <main role="main">

            <title>About</title>

            <h1>About me</h1>

            <p>Hi there, this is some information about me.</p>

        </main>
    )
}

export default AboutPage

Once saved, your development server will rebuild the site and your About page will be available at: http://localhost:8000/about

Pages created in the src/pages directory use the filename of each page as the route. For example, if we have a page component named blog.js, the route for that page would be http://localhost:8000/blog

Components

A component is a reusable piece of the UI (User Interface) we can use within our page components, such as Header, Footer, Navigation, Button, Link etc.

Unlike page file names, Component file names always start with an uppercase letter.

Links

Gatsby has a built-in Link component. This is a pre-built component that you can use in your site and is defined and maintained by the Gatsby package, so we can import and use it within our components easily.

The Link component lets you link to another page in your Gatsby site similar to how you use an anchor tag of <a> in HTML. The Link component takes a prop called to which is similar to the href attribute of a link. The value within this prop should be the URL path to the page you want to link to.

To link to external links, URLs and/or pages not created by your Gatsby site, use the regular HTML anchor <a> tag, not the Link component.

Begin by firstly importing the Link component into your page component from the Gatsby package.

    

import { Link } from 'gatsby'

Once imported, the Link component can be used on the page.

    

<Link to="/about">About</Link>

We can add a link to the Homepage to allow us to navigate to the About page.

    

import * as React from 'react'
import { Link } from 'gatsby'

const IndexPage = () => {

            <Link to="/about">About</Link>

            <p>I'm making this by following a Gatsby Tutorial.</p>

        </main>
    )
}

export default IndexPage

And a link added to the About page to navigate back to the Homepage.

Reusable Layout Component

On most websites you may notice that there are a few repeating parts of the UI (User Interface) across all pages, such as the page title and navigation. This could be copied to each page individually, but if you then want to make changes, each page would then need to be updated. It's therefore better to create one common layout component that groups shared elements together so they can be reused. That way, you can make changes in one place and it will apply to every page.

Examples of Layout components can be Header and Footer.

    

// src/components/Footer.js

import React from 'react';

const Footer = () => {
    return (
        <footer>
            <p>&copy; Slick's Slices {new Date().getFullYear()}</p>
        </footer>
    );
}

export default Footer;

In the Layout component, we can then import the Footer component.

    

// src/components/Layout.js

import React from 'react';
import Nav from './Nav';
import Footer from './Footer';

const Layout = () => {
    return (
        <>
            <Nav />

            <p>I am the page content</p>

            <Footer />
        </>
    );
}

export default Layout;

In Gatsby, this is known as a building-block component.

For the Layout component we will be using a special React prop called children. When you render a component, the children prop will automatically be passed whatever content comes between the opening an closing tags for that component.

Begin by creating a new directory in src named components.

Within this newly created components directory, create a file named layout.js and add the following code to create our Layout component.

    

import * as React from 'react'
import { Link } from 'gatsby'

const Layout = ({ pageTitle, children }) => {
    return (
        <main role="main">

            <title>{pageTitle}</title>

            <nav>
                <ul>
                    <li><Link to="/">Home</Link></li>
                    <li><Link to="/about">About</Link></li>
                </ul>
            </nav>

            <h1>{pageTitle}</h1>

            {children}

        </main>
    )
}

export default Layout

The Layout component will render a dynamic page title using the pageTitle prop to pass the page title from the page component. Content for each page will be passed in using the children prop.

You may have noticed that the Layout component uses a slightly different syntax for its props. Instead of looking like this:

    

const Layout = (props) => {
    ...
}

Our code looks like this:

    

const Layout = ({ pageTitle, children }) => {
    ...
}

This is a JavaScript technique called destructuring and is a shortcut for defining variables based on an object's properties. It's a short way of writing this:

    

const Layout = (props) => {
    const pageTitle = props.pageTitle
    const children = props.children
}

We can now update the Homepage component to use the Layout component instead of hard-coding the Link component used earlier.

Notice how the pageTitle is being passed into the Layout component as a prop and the <p> is passed into the Layout component children prop.

    

import * as React from 'react'
import Layout from '../components/layout'

const IndexPage = () => {
    return (
        <Layout pageTitle="Home page">
            <p>I'm making this by following the Gatsby Tutorial.</p>
        </Layout>
    )
}

export default IndexPage

Let's update the About page component to also use the Layout component.

    

import * as React from 'react'
import Layout from '../components/layout'

const AboutPage = () => {
    return (
        <Layout pageTitle="About page">
            <p>Hi there, this is some information about me.</p>
        </Layout>
    )
}

export default AboutPage

What are Layout components?

Layout components are for sections of your site thay you want to share across multiple pages. For example, Gatsby sites will commonly have a Layout component with a shared header and footer. Other common things to add to layouts are a sidebar and/or navigation menu.

It is recommended to create your Layout components alongside the rest of your components - e.g. src/components/

The gatsby-browser.js file

The gatsby-browser.js file is placed in the root of your project folder, not inside the src directory.

https://www.gatsbyjs.com/docs/reference/config-files/gatsby-browser/

This is a file that lets us hook into different APIs of Gatsby if we need to. For instance, we could use the wrapPageElement hook from the Gatsby API to wrap all our pages with Layout component. Gatsby will automatically do that for us so we then don't need to do that manually on each page.

Gatsby will check this file and run any functions within by default when the site is built or published. You may need to restart your development server for the changes to take effect.

    

// gatsby-browser.js

import React from 'react';
import Layout from './src/components/Layout';

export function wrapPageElement({ element, props }) {
    return <Layout {...props}>{element}</Layout>
};

This only works in the browser, so we will also need to put this into a file named gatsby-ssr.js for server-side rendering.

    

// gatsby-ssr.js

import React from 'react';
import Layout from './src/components/Layout';

export function wrapPageElement({ element, props }) {
    return <Layout {...props}>{element}</Layout>
};

    return (
        <>
            <p>Hi there, this is some information about me.</p>
        </>
    )
}

export default AboutPage

Styling

Gatsby will automatically load your critical CSS into your pages. This helps to reduce render-blocking which can slow a site down and avoids the FOUC (Flash Of Unstyled Content).

You could begin by creating a simple CSS file in ths src/styles directory, add a CSS stylesheet and then import the CSS stylesheet into the gatsby-browser.js file to create a global style.

    

// gatsby-browser.js

import React from 'react';
import Layout from './src/components/Layout';

import './src/styles/main.css';

export function wrapPageElement({ element, props }) {
    return <Layout {...props}>{element}</Layout>
};

Gatsby will use this to apply styling as part of the first render and stick this into the HTML that renders to avoid render blocking. You won't see it in the HTML source in development mode, but you will once the site has been built for production.

Gatsby isn't strict about what styling approach you use. By default, Gatsby uses CSS modules to style components. This means that the styles are scoped to the components helping to avoid class name collisions between components.

Gatsby is automatically configured to handle CSS modules without any extra setup.

To define styles using CSS Modules, add your CSS into a file that ends with the file extension of .module.css which will then tell Gatsby that this is a CSS Module and not plain CSS.

Within your CSS Module file, create separate CSS classes for each element you want to style, as this example shows:

    

// src/components/my-component.module.css

.title {
    color: blue;
    font-size: 3rem;
}

As an example, here's a template for a React component:

    

// src/components/my-component.js

import * as React from 'react'
import { title } from './my-component.module.css'

const MyComponent = () => {
    return (
        <h1 className={title}>
            Example title styled by a CSS Module
        </h1>
    )
}

export default MyComponent

If you open the developer console in your browser and inspect the <h1> element, you'll see it has a long classname like: my-component-module-title---2lRF7. This is the class name generated by CSS Modules and is unique across your site. Even if you use the same class name on another component within another CSS Module file, it won't conflict. Using CSS Modules lets you write CSS that's scoped to your component so you don't need to worry about naming conflicts between components.

We add the classes from our CSS Module files into our React component elements using the className attribute.

Gatsby will also automatically convert your CSS class names into camel-case variables that you can then import into your JavaScript components.

You can add multiple classes into your components using CSS Modules.

    

// src/components/my-component.module.css

.title {
    color: blue;
    font-size: 3rem;
}

.heading-text {
    color: #efefef;
}

.copy {
    padding: 20px 0;
}

Our component may look something like this:

    

// src/components/my-component.js

import * as React from 'react'
import { 
    title,
    headingText,
    copy
} from './my-component.module.css'

const MyComponent = () => {
    return (
        <h1 className={title}>
            Example title styled by a CSS Module
        </h1>

        <div className={copy}>
            <p className={headingText}>Lorem ipsum text</p>
        </div>
    )
}

export default MyComponent

To set a reset/normalize and/or global styling, just import the CSS stylesheet into your Layout component:

    

import 'normalize.css';

In the example above, this has been installed as an NPM dependency package, so it doesn't require a file path.

Plugins

New functionality can be added to your Gatsby site without needing to built it from scratch yourself by using Gatsby plugins. a Gatsby plugin is a prebuilt package you can install and add to your site.

eCommerce and payment plugins, like Stripe, are available.

In Gatsby terms, a plugin is a separate NPM (Node Package Manager) package that you install to add extra features to your site. You don't need to use plugins - you could build the same functionality from scratch yourself - but they save you time. Examples include:

  • Pre-built components
  • Analytics
  • Pull data into your site

Some plugins are built by Gatsby employees, others are built and maintained by community members.

Browse the Gatsby Plugin Library to view all the available plugins.

To add a plugin to your Gatsby site:

  • Install the plugin using NPM.
  • Configure the plugin in your site's gatsby-config.js file.
  • Import the plugin into your component(s). Check the README file of the plugin to explain the setup process.

Begin by installing the plugin using NPM. In your terminal, run the following command (swapping out plugin-name for the name of the plugin you want to use). This will add the plugin as a dependency in your package.json and package-lock.json file.

bash -- 85x24

    $ npm install plugin-name

Next, configure the plugin in your gatsby-config.js file. This file contains information about your site, including configuration for plugins. You can add a plugin to your plugins array.

    

module.exports = {
    siteMetadata: {
        title: "My First Gatsby Site",
    },
    plugins: ["plugin-name"],
};

Some plugins may require extra configuration options. In that case, you'll need to add an object to the plugins array.

    

module.exports = {
    siteMetadata: {
        title: "My First Gatsby Site",
    },
    plugins: [
        {
            resolve: "plugin-name",
            options: {
                // Check the plugin README file for what options go here
            }
        }
    ],
};

After updates are made to the gatsby-config.js file, you'll need to restart the development server for your changes to take effect.

Now that you've set up the plugin, you can use it in your Gatsby site as needed.

Performance

Lighthouse/PageSpeed Insights and Webpagetest are the most common website performance testing tools in the Gatsby community. Lighthouse tends to be seen as more "canonical", Webpagetest tends to be seen as more precise.

There tools measure what are known as Core Web Vitals (CWV), which measure both time to page load and time to page interactivity.

Site performance becomes increasingly more important as your project grows. There are a few options for measuring performance:

  • Gatsby Cloud - has Lighthouse performance reports built into its CI/CD. Each time a merge to the main branch occurs in GitHub, a Lighthouse report will run and the performance score will be displayed.
  • Use third-party tools/services such as Calibre or Speedcurve.
  • Integrate Webpagetest or Lighthouse reports into your own CI/CD process, via webhooks or GitHub Actions.

If your font file is over 50kb, it's too large. In addition, fonts block page load, so it's important to think about reducing network calls.

  • Use woff2 as this is a compressed font format supported by a large majority of browsers. A few legacy browsers use woff. Consider also using image formats such as webp instead of png or jpg to reduce the amount of data sent over the network.
  • Self-host font files rather than installing from an external CDN (Content Delivery Network). Having the font file available locally will save a trip over the network and reduce blocking time.

Headless CMS

A headless CMS (Content Management System) enables writers and content editors to produce and organise content, whilst providing developers with the structured data that can be displayed using a separate system on the frontend of a website or app.

Traditional, monolithic CMS platforms are responsible for both the backend management of content and serving the content to the end user. A headless CMS is decoupled from the frontend, meaning the developers can build great experiences for the end user using the best technologies available. Many traditional CMS platforms such as WordPress and Drupal, now support a "headless" or "decoupled" mode, which is ideal for Gatsby sites. Through the use of source plugins, Gatsby has support for many headless CMS options, allowing your content team to continue to use a familiar and/or preferred admin interface.

If your preferred CMS integration with Gatsby is not available, you can open a pull request to add it. Alternatively, you can write your own source plugin.

GraphQL & Getting Data

So far, everything added to our Gatsby site has been added directly into our React components. It's often easier if the data used in the site can be created and maintained somewhere else - like a folder of Markdown files, or from a Content Management System (CMS) - and then pulled into our components as needed. That way, we can make updates to our site content without affecting or changing the code of your site.

Gatsby has a feature called the data layer that you can use to pull data into your site from anywhere, for instance:

  • You could have your blog posts in a headless WordPress CMS.
  • Your products for your store could be in Shopify.
  • User data held in Airtable.
  • Add JSON from a payment processor.

With the data layer, you can combine data from multiple sources.

First, your data is stored in one or more sources. That source maybe a folder on your computer, a Content Management System (CMS) like WordPress, or a database. As mentioned, you can have multiple data sources.

This data layer is powered by a technology called GraphQL, a query language with a special syntax that lets you ask for the data you need from inside a component.

Gatsby keeps all the data for your site in the GraphQL data layer.

Getting data from the source (Source Plugins)

To get data from its source (such as from a CMS) into the data layer, we need to add a type of plugin to our Gatsby site called a source plugin. Each source plugin is designed to communicate with a specific source. When you build your site, each source plugin pulls data from its particular source and adds it to your site's GraphQL data layer.

You can identify source plugins by their name. They typically start with gatsby-source-. Examples include gatsby-source-filesystem and gatsby-source-contentful.

You can write GraphQL queries inside your components to pull data from the data layout into your Gatsby site. When you build your site, Gatsby will find all the GraphQL queries in your components, run them, and put the resulting data into your component to render.

Explore the Data Layer using GraphiQL

When you start the local development server for your site, Gatsby automatically create a special endpoint that lets you use an in-browser tool called GraphiQL. This tool can be used to expore your site's data layer and build GraphQL queries. The endpoint will be located in your terminal output once the build has completed (it has three underscores in the URL), as below:

http://localhost:8000/___graphql

GraphiQL is a helpful tool for testing out your GraphQL queries before you add them to your code. This allows us to make sure our queries always respond with the data you expect.

The process for using GraphQL queries can differ slightly depending on whether you're using a page component or a building-block component.

If you look inside the gatsby-config.js file, you'll see that information about your site has been added to the siteMetadata object. This data was added to the gatsby-config.js file automatically when you used the gatsby new command. This data also gets pulled into the GraphQL data layer automatically, so we don't need a source plugin.

    

module.exports = {
    siteMetadata: {
        title: "My First Gatsby Site",
    },
    plugins: [
        ...
    ],
};

You don't want to place sensitive information into your gatsby-config.js file because it gets added to your version control. So don't include passwords or tokens from APIs into this file. Use a .env file for this that is added to the root of your project folder.

    

SANITY_TOKEN=2t3bjbeibeieh73gbsbjdbsdbsbdbsdbjsdh

We can then access this. You'll need to install the dotenv package and import it into the gatsby-config.js file.

    

import dotenv from 'dotenv';

dotenv.config({ path: '.env' });

module.exports = {
    siteMetadata: {
        title: "My First Gatsby Site",
    },
    plugins: [
        resolve: 'gatsby-source-sanity',
        token: process.env.SANITY_TOKEN,
    ],
};

Next, open the GraphiQL tool as we now want to write a GraphQL query to query the data in the data layer (assuming the development server is running).

http://localhost:8000/___graphql
  • Within the site field, open the second dropdown for the siteMetadata (the blue one). This corresponds to the siteMetadata object in the gatbsy-config.js file.
  • The purple one with a colon, is actually an argument attached to the site field that can be used to filter which pieces of data from the data layer get returned in the response, but the blue one without the colon is what we will commonly use as this is the actual siteMetadata field which tells GraphQl what to include in your response data.

Within siteMetadata, check the box next to the title field. The query in your query editor should look like this:

    

query MyQuery {
    site {
        siteMetadata {
            title
        }
    }
}

Click the Execute Query button (the "play" triange at the top of the page) to run the query.

The JSON (JavaScript Object Notation) response should look something like the object below. The structure of the data object in the response should match the structure of the fields in the query.

    

{
    "data": {
        "site": {
            "siteMetadata": {
                "title": "My First Gatsby Site"
            }
        }
    },
    "extensions": {}
}

If you now change the title property in the gatsby-config.js file, when you save the file, rebuild and re-run the query, GraphiQL will show the update data (title).

Adding data to Components

Now that the GraphQL query is returning data from the data layer, we can now pull the data into our React components. To pull data into a building-block component, such as Layout, we will use a pre-defined Gatsby function called useStaticQuery. This is a pre-defined hook from the Gatsby package that lets you add GraphQL queries to your building-block components.

The useStaticQuery function takes one parameter, a templated string of the GraphQL query you want to run, and returns the requested data, which you can store in a variable and then use throughout your component.

We begin by importing the useStaticQuery hook and graphql tag from the gatsby package into your React component. The graphql tag is something called a tagged template literal and this tells Gatsby that the string following it is a GraphQL query so Gatsby can parse and run the query to get the data from the data layer.

    

import { useStaticQuery, graphql } from 'gatsby'

Inside your React component, call useStaticQuery using the graphql template tag and your query from GraphiQL. This stores the results in a variable so that you can use it later in your component.

    

const data = useStaticQuery(graphql`

    // Copy and paste your query from GraphiQL here and delete the query name "MyQuery"

`)

By default, the query you build in GraphiQL will have a query name likeMyQuery. You may see an error if you have more than one query with the same name, so after you copy your query over from GraphiQL to your component, delete the name.

We can then use the data in our React component by using the dot operator (.) to access the appropriate field from the response.

    

<h1>{ data.site.siteMetadata.title }</h1>

You can only call useStaticQuery once per file. If you need multiple fields, you can add them all into a single query.

For example, if you need data from both the site field and the siteBuildMetadata field, you could make the following call to useStaticQuery:

    

const data = useStaticQuery(graphql`
    query {
        site {
            siteMetadata {
                title
            }
        }
        siteBuildMetadata {
            buildTime
        }
    }
`)

Let's update the Layout component to pull data into the site title from the data layer using useStaticQuery.

    

import * as React from 'react'
import { Link, useStaticQuery, graphql } from 'gatsby'

const Layout = ({ pageTitle, children }) => {

    const data = useStaticQuery(graphql`
        query {
            site {
                siteMetadata {
                    title
                }
            }
        }
    `)

    return (
        <main role="main">

            <title>{pageTitle} | {data.site.siteMetadata.title}</title>

            <nav>
                <ul>
                    <li><Link to="/">Home</Link></li>
                    <li><Link to="/about">About</Link></li>
                </ul>
            </nav>

            <h1>{pageTitle}</h1>

            {children}

        </main>
    )
}

export default Layout

Once the browser reloads, you should see your changes. We have successfully used GraphQL to pull data from the data layer into your site. If you change the site title in your gatsby-config.js file, you'll see your site update in the browser.

This example is for non-page components, like Layout, that are used within the page components to build the UI.

Looping over Data

Begin by creating a list component in the src/components directory and importing the list component into one of your page components created.

The page query can be passed down into the List component via props. Imagine this is our page component:

    

import * as React from 'react'
import { graphql } from 'gatsby'
import Layout from '../components/layout'

const ComponentPage = ({ data }) => {

    return (
        <List test = {data.test.nodes} />
    )

We can then go into the List component and destructure the prop. We can use a map() function to loop over the data. We use this function to loop over something in React.

    

import React from 'react'

const List = ({ test }) => {

    return (

        <>
            {test.map((singletest) => (
                <p>{singletest.name}</p>
            ))}
        </>

    )

We could also use a seperate function (component) to render out the data.

    

import React from 'react'

const SingleItem = ({ singletest }) => {
    return <p>{singletest.name}</p>;
}

const List = ({ test }) => {

    return (

        <>
            {test.map((singletest) => (
                <SingleItem test={test} />
            ))}
        </>

    )

When looping through the data, each child in the list should have a key prop. This helps React to know which item to re-render if needed.

    

<>
    {test.map((singletest) => (
        <p>key={singletest.id} {singletest.name}</p>
    ))}
</>

Build (Production)

Once the initial development of your Gatsby site is complete, you will need to create a production build. The Gatsby development server is optimised for making development fast, but the site generated isn't optimised for production.

To create a production build, stop the development server (if running) and run:

bash -- 85x24

    $ gatsby build

This does a production build of your site and outputs the built static files into the public directory. Once this starts, you can view your site at: http://localhost:9000

Once a production build of your site is generated, you can then use Lighthouse to audit it. If you use Gatsby Cloud, a Lighthouse audit runs automatically every time the site builds. Using the Gatsby Cloud, every code commit triggers a build, allowing you to see an shifts in your Lighthouse audit scores and what code changes caused them.

Performance is likely to be very good for your Gatsby site, but you may need to make additions to for things such as PWA (Progressive Web App), Accessibility, Best Practices, and SEO (Search Engine Optimisation) to improve your scores.
