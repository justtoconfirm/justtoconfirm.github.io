Release & Deployment

CI and CD are two acronyms that are often mentioned when we talk about modern development practices. Continuous Integration (CI) is the practice of making and preparing a release easier, CD can either mean Continuous Delivery or Continuous Deployment.

Continuous Integration (CI)

Continuous Integration (CI) is the process of automating the build and testing of code every time a team member commits changes to version control. Committing code triggers an automated build system. CI has emerged as best practice to allow developers to integrate their changes with the rest of the team's code base. This has a few benefits:

  • Reduces potential merge conflicts as code is merged once completed.
  • Reduces bugs by catching them earlier in the development cycle.
  • Automated tests run for every build to ensure builds maintain a consistent quality. Automated tests are run whenever a new commit is merged or integrated into the main branch.

Developers that practice CI merge changes back to the main branch as often as possible so the changes can be validated and automated tests can run againts the build. This helps to avoid the "integration hell" that usually happens when developers wait to merge changes into a release branch.

The process - using a modern version control system such as Git:

  • A developer will complete a code change on a feature branch and submit a "pull request" (PR) once the feature is complete.
  • On approval of the PR, the changes get merged into the master branch.
  • The developer can then delete the feature branch.
  • Build definitions are created to ensure that every commit to the master branch triggers the automated build and testing process.

Continuous Delivery (CD)

Continuous Delivery (CD) is the process to build, test, configure and deploy from a build to a production environment.

It is an extension of CI to make sure you can release new changes to your users quickly. This means, that on top of having automated your testing, you've also automated your release process and can deploy your application at any point of time by clicking on a button.

With Continuous Delivery (CD), you can therefore decide to release when you want (daily, weekly, or whatever suits your needs). Ideally you should deploy to production as early as possible to make sure that you release small batches that are easy to troubleshoot in the case of problems.

Multiple testing or staging environments create a Release Pipeline to automate the deployment of a new build. Continuous Integration (CI) starts the CD process and the pipeline stages each successive environment to the next upon successful completion of tests.

This process helps to remove the previous manual processes that led to unreliable releases that can produce delayes and errors.

Continuous Deployment (CD)

Continuous Deployment (CD) goes one step further that Continuous Delivery. With this practice, every change that passes all stages of your production pipeline is released to production. There's no human intervention and only a failed test will prevent a new change from being deployed.
