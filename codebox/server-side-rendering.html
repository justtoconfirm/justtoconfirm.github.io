Server-side Rendering (SSR)

Server-side rendering (SSR) is a technique for rendering a client-side Single page application (SPA) on the server and sending a fully rendered page to the client. This allows for dynamic components to be served as static HTML markup.

This approach can be useful for Search Engine Optimisation (SEO) when indexing does not handle JavaScript properly.

JavaScript currently enables you to build three types of applications:

  • Single Page Applications (SPA)
  • Static-generated (or pre-rendered) sites
  • Server-side rendered (SSR) applications

Servers

NodeJS is not available as an Apache module in the way mod-perl and mod-php are, so it's not possible to run Node on top of an Apache server.

Single page applications (SPA)

SSR is a often used when building Single page applications (SPA). SPAs comes with many challenges, one of which is Search Engine Optimisation (SEO).

To resolve these issues we can make use of a static site generators or server-side rendering (SSR).

Types of SSR languages or frameworks include: PHP, Java, .Net and Node.js

React & Server-side Rendering (SSR)

With SSR, the initial page is rendered from the server, meaning the subsequent pages load directly from the client. This provides React applications with the benefit of the initial server-side content and improved speed of subsequent page loads.

The benefits of a Server-side rendered React application are:

  • Performance
  • Search Engine Optimisation (SEO)
  • Social sharing

Alternatives to Server-side Rendering (SSR)

  • Nuxt.js - server-side rendering framework for Vue.js.
  • Gatsby - React-based Static Site Generator. SSR is not at runtime, rather, Gatsby does server-side rendering with Node.js at build time, creating the static HTML, CSS & JavaScript when deploying the site. This leads to fast load times and route-based code splitting and prefetching.
